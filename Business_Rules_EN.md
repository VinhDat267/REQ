# Business Rules & Workflow Overview
## Wonton POS System - Wonton Noodle Shop

This document defines the core business rules and realistic operational workflows for the wonton noodle shop. It serves as the logical foundation for translating business needs into Use Cases, database design, and UI/UX flows.

> **Business scope lock (2026-04-17):** These business rules are locked as the final baseline. From this point forward, work may clarify wording, keep VN/EN documents aligned, or map the rules into BRD/SRS/UC/diagram artifacts without changing business behavior. Any new scope expansion requires an explicit request and a scope-change ADR.
>
> **Scope-change ADRs approved as of `2026-04-17`:** `ADR-023` (Comp status, §3n), `ADR-024` (Forfeited status, §3c), `ADR-025` (Reorder data drift, §2a), `ADR-026` (Shift-span attribution, §3g). The rules introduced by these ADRs have been integrated into this baseline.

### 1. Service Models
- **Dine-in:**
  - Customers order via QR code at the table or directly at the counter.
  - **Table QR code:** Each table has a fixed QR code (printed sticker on the table) generated by the Manager via the Table Management feature. The QR encodes a URL in the format `{domain}/order?table={table_id}`. When scanned, the Client WebApp automatically links the order to that table without the customer needing to enter a table number manually. If a customer scans the QR of table A but sits at table B (wrong scan), a Server or Cashier can transfer the order to the correct table via the Table Transfer function (UC-12).
  - **Table-session validation:** QR speeds up table identification, but it does not replace on-floor staff validation. FOH staff must still be able to confirm or correct the assigned table when customers change seats, scan the wrong QR, or when different parties rotate through the same physical table.
  - "Eat first, pay later" is permitted (unless they choose to pay immediately at the counter).
  - Each dine-in order must be linked to a specific Table ID on the floor plan so food is delivered to the correct table.
  - **Additional orders (Multi-order per table):** When a dine-in customer wants to order more items after the initial order, the system creates a new order linked to the same table. A table may have multiple concurrent orders. At payment time, the cashier can see all unpaid orders on that table and may settle them all at once or individually.
  - **Transferring a table that has multiple orders:** When the Cashier performs a table transfer (UC-60), the system by default moves **all active orders of the source table** to the destination table. If only a subset needs to be moved (for example, a party splits off to a separate table), the Cashier selects the specific orders from the source table's order list. The source table returns to `Available` only when all its orders have either been transferred away or settled.
- **Takeaway:**
  - Customers may order at the counter or create a takeaway order through the Client WebApp.
  - If the customer orders directly at the counter, they **may pay immediately at the counter** using cash, QR, or another supported method handled by the Cashier.
  - If the customer creates the order through the Client WebApp, payment is completed through the online / bank-transfer-confirmation flow before the order enters operations.
  - The mandatory rule for Takeaway is 100% payment **before** the order is printed or pushed to the kitchen. "Receive the bag first, pay later" is not allowed for Takeaway.
  - The kitchen packs the food and places it in the designated Takeaway waiting area.
  - **Order identification at handoff:** The kitchen ticket prints a short order code (e.g., last 4 characters). The Cashier calls the customer's name or matches the order code on the POS screen with the code on the bag to hand over the correct order. For Pickup orders, the same process applies — the customer shows the order code on their phone or reads it aloud.
- **Pickup (Scheduled via WebApp):**
  - Customers order online and select a precise pickup time without waiting in line.
  - Requires 100% upfront online payment to prevent abandoned orders.
  - Timeslots must fall within operating hours and pass capacity validation against kitchen workload.
  - **Pickup capacity configuration:** The Manager sets the maximum number of Pickup orders allowed per time slot via the Pickup Settings section in the Admin WebApp. For example: a maximum of 5 Pickup orders per 30-minute slot. In the current final baseline, capacity is modeled as a **simple order-count-per-slot approximation**, not a weighted kitchen-load formula per item; therefore, the Manager is expected to reduce limits or toggle "Pause Pickup" when the kitchen is overloaded, understaffed, or handling several large orders at once. When a slot reaches its limit, the system prevents customers from selecting it and suggests nearby available slots.
  - **Impact of capacity changes / pause toggle:** Reducing the limit or enabling "Pause Pickup" applies only to **new Pickup orders**. Pickup orders that were already auto-accepted before the change retain their service commitment; if the shop cannot fulfil them, it must handle them through the Pickup exception path (reschedule / substitute / refund) under Manager authority.
  - If the slot is valid and capacity is available, the system auto-accepts the Pickup order; no manual manager approval is required.
  - **Kitchen must not reject a paid Pickup order:** Because a Pickup order has been fully prepaid and auto-accepted by the system, the kitchen staff cannot reject it. If the kitchen is unable to fulfil the order (e.g., a key ingredient runs out mid-shift), the kitchen must escalate to the Manager for exception handling (substitute an equivalent item, reschedule the pickup slot, or process a refund). Only the Manager has the authority to cancel or modify a paid Pickup order.

### 2. Account & Role Rules (RBAC)
- **Guest Checkout:**
  - Customers are not forced into a full registration flow to place an online order.
  - A guest order only requires `{Name + Phone Number}` at checkout.
  - After the order is created, guest customers may track it via `{Order Code + Phone Number}` without logging in.
  - **Guest order code security:** The Order Tracking Code must be sufficiently long and random (minimum 8 alphanumeric characters) to prevent brute-force guessing. The system must enforce rate-limiting on the guest order lookup API (maximum 5 failed attempts per IP within 15 minutes) to prevent unauthorized access.
  - **Guest personal data retention and anonymization:** Guest Name and Phone Number are stored with the order to support lookup, follow-up on issues, and financial audit. After **12 months** from when the order reaches a terminal state (`Completed` / `Cancelled` / `Refunded` / `Write-off`), the system anonymizes personal fields (replacing phone/name with anonymized values) and keeps only the depersonalized record for statistical use. This rule follows Vietnam's Decree 13/2023/ND-CP on personal data protection.
- **Registered Customers:**
  - In the full-system scope, they have full account features: login, order history, reorder, profile updates, password change, and order rating.
  - In the final-project scope, **reorder** is part of the active baseline through `UC-18` in the 74-UC inventory and must be specified in the final UC package.
- **Multi-Role Staffing:**
  - A single staff account is not practically limited to one role.
  - Managers grant privileges via multiple checkboxes such as `[x] Cashier`, `[x] Server`, `[ ] Kitchen`.
  - The UI must adapt to the union of all granted permissions.

### 2a. Reorder Rules (UC-18)
*(ADR-025)*
- **Who can Reorder:** Registered Customers only. Guests have no account-linked order history to reorder from.
- **Eligible source orders:** Only orders in `Completed` state. Orders in `Cancelled`, `Refunded`, `Write-off`, `Comp`, or `Forfeited` are hidden from the Reorder list because the customer never actually received that meal.
- **Handling menu drift between original order and reorder time:**
  - **Item deleted or archived:** drop from the reorder cart; show inline notice "{item_name} is no longer on the menu and has been removed from the cart".
  - **Price changed (higher or lower):** use the **current** price, not the historical one. Inline notice "{item_name} price changed: {old} → {new}".
  - **Topping or option no longer offered:** drop the missing topping only; keep the base item. Inline notice.
  - **Item currently 86'd:** keep in cart but flagged unavailable; block checkout until the customer removes it or 86'd is cleared.
  - **Category reorganized:** invisible to the customer — reorder uses item IDs, not category paths.
- **Drift notice UX:** Per-line inline notices plus a consolidated banner at the top of the reorder review screen (e.g., "3 changes from your original order"). No blocking modal to avoid friction when many items drift.
- **Never silently change the final total:** any drift affecting the total must be visible before checkout.
- **Preserve special notes:** Reorder carries forward customer-entered notes (e.g., "less spicy") onto new cart lines when the item is still available.
- **New order, not resurrected:** Reorder creates a brand-new `order_id`; it does not revive the original order's payment or state.
- **All items dropped case:** if drift leaves no available items, show "This order has no available items" with a link back to the menu.
- **Retention:** Registered Customer history is not subject to the 12-month Guest anonymization rule in §2. Reorder works on any `Completed` order still in the account's history.

### 3. Cancellation & Financial Rules
- **Separate order and payment states:**
  - `order_status` reflects operational progress (`Pending Confirmation -> Cooking -> Ready -> Completed / Cancelled`).
  - `payment_status` reflects financial progress (`Pending Online Payment -> Paid` for online / bank-transfer-confirmation flows, or `Unpaid -> Paid` for Dine-in and selected counter-collection flows).
- **Auto-cancellation:**
  - Orders that require prepayment **placed through the Client WebApp** (`Pickup` and `Takeaway online`) must automatically transition to `Cancelled` if payment is not completed within 15 minutes.
  - **Exception for counter Takeaway:** When a Cashier creates a Takeaway order at the counter (the customer is physically present), the system must not auto-cancel after 15 minutes. The Cashier may collect payment immediately at the counter or keep the order briefly pending while the customer is still present; if the customer abandons the purchase, the Cashier cancels it manually.
- **Takeaway must be paid before kitchen intake:**
  - Counter staff or the Client WebApp may only push a Takeaway order to the kitchen when `payment_status = Paid`, regardless of whether payment was collected immediately at the counter or through an online flow.
- **Dine-in auto-cancel exemption:**
  - Unpaid Dine-in orders must never be auto-cancelled.
  - Instead, they remain visibly pinned as `Completed + Unpaid` on the cashier dashboard until staff settle the bill.
  - **Late-payment escalation:** If a Dine-in order remains `Completed + Unpaid` for more than 30 minutes, the system automatically sends a push notification alert to the Manager with the table number and outstanding amount.
  - **Dine-and-dash handling:** If a customer has left the premises without paying, the Cashier marks the order as `Write-off` with a reason. The `Write-off` action requires Manager confirmation. `Write-off` is a manager-approved loss-recording state, **not** a successful customer payment outcome. Write-off orders are included in the end-of-day loss report for Manager review.
- **Practical cancellation matrix:**
  - `Dine-in + Unpaid + not yet accepted by kitchen`: FOH/Cashier may cancel to fix the order or because the customer changed their mind.
  - `Takeaway/Pickup + Pending Online Payment`: the customer may abandon payment; Client WebApp orders auto-cancel after 15 minutes, while counter-created Takeaway orders are cancelled manually by the Cashier if the customer walks away.
  - `Paid + not yet accepted by kitchen`: the customer may request cancellation, but only the Manager may approve the refund.
  - `Cooking` or `Ready`: the customer cannot self-cancel directly; only the Manager can handle the exception path (substitution, refund, or loss handling depending on the situation).
- **Customer abandonment after ordering:**
  - **Counter Takeaway, still unpaid:** the Cashier cancels manually. The order must not reach the kitchen. If staff mistakenly started production before collecting payment, the Manager confirms the cancellation reason and records it as an internal operational loss, not revenue.
  - **Dine-in, not yet served and still unpaid:** FOH/Cashier may cancel manually if the customer leaves shortly after ordering or changes their mind before the kitchen actually starts the dish.
  - **Dine-in, already served / consumed but customer leaves without paying:** handle through the manager-approved `Write-off` path.

### 3a. QR Payment Rules
- **Two distinct QR payment types:**
  - **Payment Gateway QR (MoMo / VNPay / ZaloPay):** The system generates a dynamic QR containing order details. The customer scans and confirms in the wallet app. The gateway sends an automatic callback → `payment_status` transitions to `Paid` automatically. This is the reliable confirmation path.
  - **Bank Transfer QR:** The system displays a QR containing the shop's bank account number and the order amount. The customer scans with any banking app and transfers funds. The system **does not receive an automatic callback** from the bank (unless a separate bank API integration exists). The Cashier must **manually confirm** receipt (by checking the shop's banking SMS/app) and then tap "Confirm Received" in the system.
- **Manual confirmation risk acknowledgement:**
  - With bank transfer QR, there is a risk that the Cashier confirms incorrectly (taps received before funds arrive, or the customer transfers an insufficient amount). The system logs every manual confirmation with timestamp and order ID for end-of-day reconciliation.

### 3b. Refund Rules
- **When a refund is triggered:**
  - When an order with `payment_status = Paid` transitions to `order_status = Cancelled` (Manager cancels, kitchen cannot fulfil, or customer requests cancellation before the kitchen accepts the order).
- **Refund approval authority:**
  - Only the Manager may approve a refund request. Cashiers and kitchen staff cannot issue refunds on their own.
- **Refund method:**
  - Refunds are processed through the same payment method used for the original transaction. For online payment gateways (MoMo/VNPay/ZaloPay), the system sends a refund request via the gateway API. For QR bank transfers, the Manager processes the refund manually and marks it as completed in the system.
- **Payment status during refund:**
  - When a refund request is created, `payment_status` transitions to `Refund Pending`. When the refund is completed (gateway callback or Manager manual confirmation), `payment_status` transitions to `Refunded`.
- **Full vs partial refund:**
  - If only **part of a paid order** cannot be fulfilled (for example, one item is unavailable, a preparation issue occurs, or the customer agrees to drop one item), the Manager may approve an equivalent substitution, remove the item and refund the difference, or cancel the entire order if the customer rejects the alternative.
- **Refund failure or delayed refund:**
  - If the payment gateway/API refund fails, or a manual bank-transfer refund has not been completed yet, the order must **not** reopen for operations. `payment_status` remains `Refund Pending` until the Manager retries or confirms completion.
- **Audit trail:**
  - Every refund transaction must be logged with the cancellation reason, approver, timestamp, and refund method for end-of-day reconciliation.

### 3c. Order Completion Rules (Ready → Completed)
- **Dine-in:** The Server taps "Delivered" after bringing the food to the correct table → `order_status` transitions to `Completed`. If a table has multiple orders, each order is marked `Completed` independently once served.
- **Takeaway:** The Cashier taps "Customer Picked Up" when the customer collects the order at the counter → `order_status` transitions to `Completed`.
- **Pickup:** Same as Takeaway — the Cashier taps "Customer Picked Up" when the customer arrives at the scheduled time and collects the order → `order_status` transitions to `Completed`.
- **Note:** `Ready` only means the kitchen has finished cooking. An order is `Completed` only when the customer has actually received the food.
- **Late / no-show Takeaway handling:** For a Takeaway order that is already `Paid` and `Ready`, if the customer does not arrive within the shop-configured holding window (suggested default 30 minutes after `Ready`, or until closing time), the system flags it as overdue for Cashier/Manager handling. The system must not auto-complete the order and must not auto-refund it. Before making a decision, the shop is **strongly recommended** to call the customer at least once using the phone number on file. The Manager then chooses one of: continue holding the food, remake if the customer arrives, full/partial refund, or `Forfeited` (shop retains payment) — see ADR-024.
- **Late / no-show Pickup handling:** Once a Pickup order is marked `Ready`, it remains `Ready` until the customer actually collects it. If the scheduled time plus the shop-configured holding window has passed (suggested default 60 minutes after the scheduled time, or until closing time), the system raises an overdue flag for the Cashier/Manager to handle. The system does not auto-complete the order and does not auto-refund it. The Manager chooses one of: continue holding, cancel with full/partial refund, or `Forfeited` (shop retains payment) after a customer-contact attempt. `Forfeited` must go through `§3k` audit (ADR-024).
- **Early Pickup arrival:** If a customer arrives before the scheduled Pickup time and the order is not yet `Ready`, the Cashier informs the customer of the remaining waiting time. The system does not promise early handoff merely because the customer arrived early; accelerating preparation is an operational decision by the shop.

### 3d. Online Payment Failure Rules
- **Multiple retries on the same order are allowed:**
  - For prepaid online orders, if one payment method fails, the customer may retry the same method or switch to another method on the **same existing order**; the system must not create duplicate orders just because the customer retries payment.
- **When all payment methods fail:**
  - The order remains in `Pending Online Payment`, must not be sent to the kitchen, and the payment screen must clearly tell the customer that they may keep retrying until the order-hold window expires.
  - If payment is still not completed within the hold window (default 15 minutes for prepaid Client WebApp flows), the system auto-transitions the order to `Cancelled`.
- **System-wide gateway outage:**
  - If multiple online payment methods fail at the same time because of a system or gateway outage, the system must show a clear message such as "Online payment is temporarily unavailable, please try again later." The order still must not enter kitchen operations until valid payment is confirmed or the hold window expires.
- **Late success callback after the order has auto-cancelled:**
  - If the system has already auto-cancelled the order because the hold window expired, but a success callback arrives afterwards, the system must **not** reopen the order and must **not** push it to the kitchen automatically. It logs the anomaly, notifies the Manager, and starts a refund flow by moving the payment side to `Refund Pending`.
- **Idempotency and duplicate transactions:**
  - Repeated success callbacks for the same transaction, or retries after the order is already `Paid`, must not create another order, another receipt, or another kitchen ticket. If true duplicate collection is detected, the Manager handles the excess amount through the refund flow while the original order remains unchanged.

### 3e. Order Modification Rules
- **Before kitchen acceptance:**
  - FOH/Cashier may edit items, quantities, toppings, or notes for Dine-in and counter-created orders while the kitchen has not started processing.
  - For paid Takeaway/Pickup orders that the kitchen has not accepted yet, a change that increases the total requires additional payment before kitchen intake; a change that decreases the total requires Manager-approved partial refund or full cancellation under the refund rules.
- **After kitchen acceptance or once cooking has started:**
  - Customers cannot directly cancel or silently edit items already in production. Only the Manager may handle exceptions such as substitution, partial refund, or remake based on the real situation.
  - For Dine-in, additional items after that point are created as a **new order** linked to the same table instead of retroactively editing the running order.
- **Operational notes vs financially-impacting edits:**
  - **Minor notes that do not change price or a main ingredient** (for example, "less spicy", "no scallions", "no ice"): FOH/Server may relay them directly to the kitchen without Manager approval, provided the kitchen confirms it can still apply them. These actions are still logged against the order for traceability.
  - **Financially-impacting changes** (swapping an item, adding/removing priced toppings, changing quantity, dropping an item): must go through the Manager-approved exception flow as stated above.
- **No silent self-edit after submission:**
  - After an order has been submitted, the Client WebApp must not be allowed to change a running order invisibly without FOH/Cashier/Manager oversight.

### 3f. Wrong Handoff / Wrong Delivery Rules
- **Incident must be reported immediately:**
  - If a Server delivers to the wrong table, the Cashier hands over the wrong bag, or staff confirms the wrong customer pickup, the incident must be flagged immediately for Manager/FOH handling.
- **Completion correction for mistaken handoff:**
  - If staff mistakenly marked the order `Completed` but the customer has **not actually received the correct order**, the Manager or FOH may revert the order from `Completed` back to `Ready` within the same shift with a required audit reason.
- **Operational consequence handling:**
  - If the wrong customer has already left with the food, the Manager decides whether to remake, refund, partially refund, or record internal loss. The system must not treat this as a normal completed transaction just because staff tapped completion too early.

### 3g. Shift Closing & End-of-Day Reconciliation Rules
- **Pre-close review requirement:**
  - Before the Cashier/Manager closes a shift, the system must warn or block final confirmation if unresolved operational/financial exceptions still exist.
- **Shift opening and cash drawer ownership:**
  - Each cashier shift must record the responsible staff member, opening time, starting cash amount, and active counter/device. Cash cannot be recorded into a shift that has already been closed.
  - At shift close, the Cashier enters the counted cash amount. The system compares it with expected cash based on paid orders, manual refunds, write-offs, and adjustments during the shift.
  - Cash differences require a reason and Manager confirmation when they exceed the configured threshold.
- **States that must be reviewed:**
  - `Pending Online Payment` that has exceeded the hold window but is not cancelled.
  - `Ready` orders that remain overdue for handoff.
  - `Refund Pending` cases not yet completed.
  - `Write-off` records waiting for Manager confirmation.
  - Wrong-handoff incidents, partial-refund differences, or duplicate-charge incidents not yet closed.
- **Reconciliation goal:**
  - End-of-shift reporting must clearly separate valid paid revenue, refunded orders, internal operational loss, and unpaid abandoned transactions so revenue figures are not distorted.
- **Reconciliation by payment method:**
  - Shift reports must separate cash, manually confirmed bank-transfer QR, online payment gateway collections, refunds, duplicate collections, write-offs, `Comp`, and `Forfeited`.
  - Bank-transfer QR must not be treated as fully reconciled unless the Cashier/Manager has recorded confirmation evidence during the shift.
- **Shift-span financial event attribution** *(ADR-026)*:
  - The baseline shift model supports **multi-shift-per-day**; a shop running a single daily shift is a degenerate case of the same model.
  - Each financial event is attributed to a shift by the **event timestamp**, not by order creation time:
    - Payment completion (`Paid`) → the shift that was open at the moment the order reached `Paid`.
    - Cash drawer movements → the shift open at the time of the movement.
    - Refund completion (`Refunded`) → the shift open when the refund **completed**, not when it was requested.
    - `Write-off` / `Comp` / `Forfeited` confirmations → the shift open at the moment the Manager confirmed it.
  - An order created in one shift but with a financial event in a later shift (for example, a Dine-in party seated at 22:30 that pays at 00:45) books the revenue in the shift that captured the payment.
- **Inherited exceptions handed over between shifts** *(ADR-026)*:
  - A shift must not be closed while it still has unresolved exception states **belonging to that shift**. If they cannot be resolved in time, the closing operator must explicitly hand them over with an audit note, and they appear on the next shift's opening screen as "inherited exceptions".
  - The carry-over list is limited to the following set, not every in-flight state:
    - `Refund Pending` not yet completed.
    - Overdue Takeaway/Pickup awaiting Manager decision (hold / remake / refund / `Forfeited`).
    - `Write-off` proposed but not yet Manager-confirmed.
    - `Forfeited` proposed but not yet Manager-confirmed.
    - Dine-in orders in `Completed + Unpaid` not yet settled.
- **Daily reporting:**
  - End-of-day reports are a pure aggregation of the day's shifts. Each event already belongs to exactly one shift, so no re-attribution logic is needed at the daily level.

### 3h. Promotion Rules
- **Promotion validity:**
  - Each promotion code must have an effective period, discount type (`%` or fixed amount), optional minimum condition, and `Active/Inactive` status.
  - The system must validate the code before payment and reject expired, inactive, invalid-condition, or usage-limit-exceeded codes.
- **Usage limits:**
  - A code may have a total usage limit, per-customer limit, or daily/time-window limit. For Guest customers, per-customer limits are checked by phone number.
- **Promotion stacking:**
  - By default, multiple promotion codes cannot be stacked on the same order. If stacking is introduced later, the stacking rule must be explicitly configured by promotion type.
- **Cancelled orders and usage recovery:**
  - If an order is cancelled before kitchen acceptance and no revenue is recognized, the system restores the promotion usage count only if the promotion policy allows it. If the order was served or only partially refunded, the Manager decides whether the usage count is restored.
- **Promotion auditability:**
  - Creating, editing, disabling, and applying promotion codes, including failed application attempts, must be logged so the Manager can review discount impact on revenue.

### 3i. Complaint / Remake Rules
- **Complaint classification:**
  - When a customer reports a missing item, wrong item, defective dish, wrong handoff, or poor experience, FOH records the issue type, related order, related item, and a short description.
- **Resolution options:**
  - The Manager or authorized FOH staff may choose remake, equivalent substitution, goodwill discount, partial refund, full refund, full-order `Comp` (proactive shop-initiated waiver, see §3n), or internal loss recording (`Write-off`) depending on the context.
- **No fake revenue for remakes:**
  - A remake must not create new revenue if it only corrects a service error from an existing order. If the customer orders extra items beyond the complaint, those items must be created as a new order.
- **Link to reviews:**
  - After a complaint is handled, a Registered Customer may still review the order within the review window; complaint data helps explain low ratings.

### 3j. Operational Outage Rules
- **KDS or kitchen printer failure:**
  - If the KDS or kitchen printer fails, FOH/Kitchen switches to temporary manual tickets. When the system recovers, staff must reconcile manually handled orders to avoid duplicate cooking or missed orders.
- **Network or payment gateway outage:**
  - Prepaid orders must not enter kitchen operations without valid payment evidence. Dine-in service may continue under the shop's internal procedure, but unsynced status must be recorded for later reconciliation.
- **Post-outage recovery:**
  - After recovery, the Manager/Cashier reviews abnormal states (`Pending`, `Ready`, `Paid but not sent to kitchen`, `Manual ticket`) and closes each case with a clear reason.

### 3k. Manager Override & Audit Rules
- **Reason-required actions:**
  - Cancelling a paid order, issuing a full/partial refund, `Write-off`, `Comp` (proactive waiver, §3n), `Forfeited` (no-show retention, §3c), applying a special discount, editing an order after kitchen acceptance, correcting `Completed`, and confirming cash differences all require a reason.
- **Required audit fields:**
  - Logs must store actor, role, timestamp, related order/item, affected amount, reason, and approver when applicable.
- **No silent audit deletion:**
  - Financial/operational audit logs must not be hard-deleted. If notes need correction, the system creates an additional record instead of silently overwriting history.

### 3l. Receipt / Invoice Rules
- **Receipt printing and reprinting:**
  - The system may print receipts after payment. Reprinted receipts must be marked as reprints or logged with the reprint count to avoid confusion.
- **Receipt cancellation or adjustment:**
  - If a receipt has been printed and the order is later refunded/cancelled, the system must keep a link between the original receipt and the adjustment/refund note.
- **VAT/tax information when requested:**
  - If the customer requests VAT/tax invoice information, the Cashier records company name, tax ID, address, and recipient email according to prototype needs. Real e-invoice integration remains an extension unless promoted.

### 3m. Inventory-Lite Rules
- **Low-stock and 86'd handling:**
  - Inventory-lite supports small-restaurant operations only: low-stock warning, 86'd item marking, and reason notes for unavailable items.
- **Manual adjustment:**
  - Manager/Kitchen staff may record manual stock adjustments for spoilage, waste, temporary restock, or end-of-day count. Each adjustment requires a reason and actor.
- **Not full procurement:**
  - The final baseline does not include supplier purchasing, payables, or automatic recipe-level stock deduction unless a later scope decision promotes those flows.
- **Relationship between 86'd and stock adjustment:**
  - `86'd` (availability flag raised by the Kitchen under §4 KDS) and `stock adjustment` (manual inventory edit by Manager/Kitchen under §3m) are **two independent mechanisms** in the final baseline. A stock adjustment down to 0 does not automatically set 86'd, and clearing 86'd does not automatically restore stock. Keeping the two in sync is an operational responsibility of staff. Bi-directional automation between them is out of the baseline and can only be introduced through a new ADR.

### 3n. Comp / Voided-by-Management Rules
*(ADR-023)*
- **When to use `Comp`:**
  - When the shop **proactively** decides not to charge an order for goodwill reasons, a serious error that already incurred cost (ruined preparation, wrong delivery that required remake), VIP treatment, or a food-safety incident.
  - Distinct from `Write-off` (customer abandoned payment unilaterally) because `Comp` is a **shop-initiated** decision.
  - Distinct from `Cancelled` (order aborted before service, no cost incurred) because `Comp` applies after food cost has already been incurred.
- **Eligible order states:**
  - Allowed on orders in `Cooking`, `Ready`, or `Completed`.
  - Not allowed on orders still in `Pending Confirmation` (kitchen has not yet accepted) — use `Cancelled` in that case instead.
- **Approval authority:**
  - Only the Manager may approve `Comp`, with a mandatory reason. Cashier and Kitchen staff cannot comp an order on their own. Follows §3k Manager Override & Audit.
- **Financial impact:**
  - If the order is **unpaid** (`Unpaid`): `payment_status` transitions directly to `Comp`; no revenue is recorded; food cost is recorded as "comp expense" on reports.
  - If the order is already **paid** (`Paid`): `payment_status` goes through `Refund Pending` → `Refunded`, and the previously collected amount is reclassified as comp expense rather than regular revenue.
- **Reporting:**
  - Comp amounts are reported on a separate line in shift-close and end-of-day reports, not aggregated with `Write-off`, `Refunded`, or regular revenue.

### 4. Kitchen Display System (KDS) Rules
- **FIFO (First In, First Out):** Orders are displayed in chronological order.
- **Batch Cooking & Grouped Updates:**
  - The kitchen should not be forced into per-bowl micro-updates.
  - The KDS must support grouped updates for identical items across multiple orders so the kitchen can work in batches.
- **86'd Items (Out-of-Stock Alert):**
  - If the kitchen runs out of broth, wontons, or other key ingredients, staff can mark the item as 86'd in one action.
  - The system must instantly lock that item across the Client WebApp and cashier screens.
- **86'd impact on existing orders:**
  - 86'd does not only block **future** orders. If the item already exists in a queued order or in a paid order that has not been fully cooked, the system must raise an order issue so FOH/Manager can decide on substitution, partial refund, or full cancellation before operations continue.
- **Kitchen-ticket deduplication:**
  - A valid order may generate only **one** logical kitchen ticket per valid payment confirmation. Repeated payment callbacks, screen refreshes, or reconnect events must not create duplicate kitchen tickets for the same order.

### 5. Review & Rating Rules
- **Who may review:**
  - Only Registered Customers may submit reviews. Guest customers cannot attach ratings to orders.
- **Review window:**
  - Reviews are allowed for **7 days** after an order reaches `Completed` status.
- **Review criteria:**
  - Three mandatory 1-5 star criteria: **Food Quality / Service Speed / Overall Experience**.
  - Text comment is optional, maximum **500 characters**.
- **Inappropriate content filtering:**
  - The system must automatically detect and hide reviews containing profanity, spam, or policy-violating content.
  - **Known limitation:** Keyword-based filtering has limited accuracy with Vietnamese due to abbreviations, teen code, and creative keyword evasion. At the prototype/demo stage, the system uses a basic banned-word list; a production version could upgrade to ML-based content moderation.
- **Limit:**
  - Each order can only be reviewed **once**. No edit or delete after submission.

### 6. Notification Rules
- **Real-time:**
  - **In-app notifications for connected clients** must arrive within **<= 3 seconds** after a status change, via WebSocket/SSE.
- **Channels:**
  - **In-app:** Popup/toast notifications on both Client WebApp and Admin WebApp.
  - **Sound:** Audible alert paired with popup, especially important for KDS and cashier POS.
  - **Browser Push:** Supported when the customer grants permission. This is a **best-effort** reminder channel for background/off-tab situations and is not covered by the strict `<= 3 seconds` SLA.
- **Notification history:**
  - All notifications are stored in history; customers can revisit them via the 🔔 icon.
- **Reconnect sync:**
  - If the client disconnects, the system must replay missed notifications in chronological order upon reconnection.
- **Notification triggers:**
  - New order created -> Notify Cashier/Kitchen.
  - Order status changes (`Cooking`, `Ready`, `Completed`, `Cancelled`) -> Notify Customer.
  - Order becomes `Ready` -> Notify Server for Dine-in, or Cashier for Takeaway/Pickup.
  - 86'd item -> Notify Cashier and lock the item on Client WebApp/POS.
  - Order cancelled -> Notify Customer + Kitchen + Cashier.
